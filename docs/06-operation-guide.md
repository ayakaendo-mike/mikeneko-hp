# 更新フロー・AI運用ガイド

運用は2トラックに分ける。担当者・使用ツール・承認プロセスがそれぞれ異なるため、混同しないこと。

```
【コンテンツ更新】担当: 社内スタッフ／日常的
 スタッフ → AI(ChatGPT/Claude)に下書き依頼 → microCMSに貼付・整形・プレビュー確認
 → 公開ボタン → Webhook発火 → 自動ビルド・自動デプロイ

【コード/構造改善】担当: Claude Code（開発）／非日常的
 Issue作成 → Claude Codeがブランチ作成・実装・PR作成
 → 人間レビュー（事実確認/表現/ブランドトーン） → 承認・マージ
 → GitHub Actionsで自動デプロイ
```

## 1. コンテンツ更新フロー（スタッフ向け・Git不要）

対象: ブログ、制作実績、お知らせ、よくある質問（すべてmicroCMS管理）

1. AIツール（ChatGPT/Claude等）に下書きを依頼する
   - 依頼時は必ずブランドトーン・ターゲット・キーワードを伝える（プロンプト例は今後本ガイドに追記）
2. 下書きをmicroCMSの該当コンテンツモデルに貼り付け、見出し・画像・タグを整形する
3. プレビューで表示崩れがないか確認する
4. **公開前チェック**（必須）
   - 事実確認（誤情報がないか）
   - 専門性・独自性の追加（AIの一般論のままにしない）
   - 誤字脱字・ブランドトーンとの一致
   - 個人情報・守秘義務に抵触する内容が無いか
5. 公開ボタンを押す → microCMSのWebhookが発火 → サイトが自動的に再ビルド・再デプロイされる

> 📌 スタッフはGit / GitHubを一切操作しない。全てmicroCMSのダッシュボード上で完結する。

## 2. コード/構造改善フロー（Claude Code運用）

対象: デザイン変更、新機能、SEOロジック、ページ構造の変更（すべてAstro/Git管理）

1. GitHub Issueにタスクを起票する（例:「サービスページにFAQセクションを追加」）
2. Claude Codeがブランチを作成し、実装・コミットを行う
3. Claude CodeがPRを自動作成する（変更内容のサマリー付き）
4. **AIはmainに直接マージしない**（最重要ガードレール）
5. 人間がレビューする（観点は [05-git-github-operations.md](./05-git-github-operations.md) 参照）
6. 承認後マージ → GitHub Actionsで自動デプロイ

## 3. AI権限範囲の段階設計

いきなり「AI社員が全部やる」を目指さず、実績を見ながら段階的に自動化範囲を広げる。

| フェーズ | AIが自律的にやって良いこと | 人間承認が必須なこと |
|---|---|---|
| 初期（現在） | ブログ下書き作成、alt属性追加、内部リンク提案、軽微な文言修正提案 | 料金・プライバシーポリシー・会社概要など信頼性/法的に関わるページの変更、本番公開の最終判断 |
| 将来（拡大後） | 軽微なコンテンツの自動公開、SEOパフォーマンスに基づく改善提案のIssue自動起票 | 引き続き法的ページ・料金・ブランドに関わる変更は人間承認 |
